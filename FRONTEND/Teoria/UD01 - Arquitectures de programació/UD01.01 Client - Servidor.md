# El Model Client/Servidor a la Web

L'arquitectura web es basa fonamentalment en el model **client/servidor**, un model de computació distribuïda on les tasques es reparteixen entre els proveïdors de recursos o serveis (servidors) i els demandants d'aquests serveis (clients).

* **El Client (Entorn Client):** És l'ordinador, telèfon intel·ligent o dispositiu de l'usuari final que executa una aplicació anomenada **navegador web** (Chrome, Firefox, Safari, Edge). El client s'encarrega de demanar els recursos, processar part de la lògica visual i renderitzar la interfície d'usuari per fer-la interactiva.

* **El Servidor (Entorn Servidor):** És un ordinador remot (o un conjunt d'ells) configurat per romandre a l'espera de peticions. Emmagatzema les dades (bases de dades), processa la lògica de negoci complexa i envia les respostes adequades al client.

# El Flux de Petició/Resposta (HTTP/HTTPS)

1. **Petició (Request):** L'usuari introdueix una URL o fa clic a un botó. El navegador genera una petició HTTP/HTTPS i l'envia a través d'Internet cap a la IP del servidor.

2. **Processament:** El servidor rep la petició, en verifica l'autenticitat, consulta la base de dades si és necessari i genera una resposta.

3. **Resposta (Response):** El servidor envia els fitxers (HTML, CSS, JavaScript, imatges o dades en format JSON) de tornada al navegador del client.
# Models d'Execució de Codi: Servidor vs. Client

A la programació web moderna, és crucial diferenciar on s'executa exactament cada línia de codi:

| Característica           | Execució en el Servidor (Backend)                                              | Execució en el Client (Frontend)                                                                            |
| ------------------------ | ------------------------------------------------------------------------------ | ----------------------------------------------------------------------------------------------------------- |
| **On s'executa?**        | En el maquinari del proveïdor (Cloud, VPS, Servidor propi).                    | En el dispositiu de l'usuari final (CPU/RAM de l'usuari).                                                   |
| **Tecnologies comunes**  | Node.js, PHP, Python, Java, C#, Go.                                            | JavaScript, HTML, CSS.                                                                                      |
| **Objectiu principal**   | Accés a bases de dades, seguretat de dades sensibles, lògica de negoci pesada. | Interactivitat en temps real, control de la interfície d'usuari, renderitzat (React).                       |
| **Visibilitat del codi** | El codi font està protegit. L'usuari mai el pot veure (només rep el resultat). | El codi font és **públic**. Qualsevol usuari pot obrir les "Eines de desenvolupador" i veure el JavaScript. |
# El Navegador Web com a Entorn d'Execució

El navegador web ja no és un simple visualitzador de documents de text; avui dia és una autèntica màquina virtual capaç d'executar aplicacions altament complexes.

Per fer-ho possible, compta amb dos components clau:

1. **Motor de Renderitzat (Rendering Engine):** S'encarrega d'interpretar l'HTML i el CSS per dibuixar la pàgina a la pantalla. Exemples: *Blink* (Chrome, Edge), *Gecko* (Firefox).

2. **Motor de JavaScript (JS Engine):** Interpreta i executa el codi JavaScript utilitzant compilació en temps real (JIT). El més conegut és el motor **V8** (utilitzat per Google Chrome i també per Node.js al servidor).

## Com coneix JavaScript l'entorn on corre? L'objecte `navigator`

Tot motor de JavaScript en l'entorn client té accés a una sèrie d'objectes globals predefinits pel navegador. Un dels més importants és l'objecte **`navigator`**. Aquest objecte actua com el "passaport" del client; conté propietats que permeten al codi saber si el dispositiu té connexió a internet (`navigator.onLine`), l'idioma del sistema (`navigator.language`) o una cadena de text tècnica anomenada **`userAgent`**, que detalla la marca, versió exacta del navegador i el sistema operatiu de l'usuari. Això permet que una web s'adapti automàticament si detecta que s'obre des d'un iPhone o un ordinador antic.
## Capacitats i Limitacions del Navegador: El "Sandbox"

Per motius de seguretat, els navegadors web apliquen un aïllament estricte anomenat **Sandbox (Entorn de seguretat o "Sorral")**. Això significa que el codi JavaScript que es descarrega d'Internet s'executa dins d'una "gàbia" virtual i té prohibit interactuar directament amb el sistema operatiu de l'usuari.

### Què POT fer JavaScript al navegador (Capacitats):

* Manipular de forma dinàmica l'estructura de la pàgina (l'arbre DOM).
* Reaccionar a les accions de l'usuari (clics, reproduir un àudio, teclejar).
* Emmagatzemar dades locals al navegador (`localStorage`).
* Realitzar peticions asíncrones (peticions de xarxa en segon pla) a APIs externes.
### Què NO POT fer JavaScript al navegador (Limitacions i el perill de `file://`):

* **Accés al disc dur:** L'accés al disc dur local està prohibit SEMPRE. Un script de JavaScript mai pot utilitzar comandes com fetch()[^1] per llegir fitxers directament del disc dur de l'usuari, independentment del protocol (http, https, file o ftp). Les peticions d'Internet només poden demanar recursos a servidors webs remots a través de la xarxa.

* **El protocol de fitxers locals (`file://`):** Quan fas doble clic a un HTML des de l'escriptori, la barra d'adreces utilitza el protocol file://. A la pràctica 1.1 veureu que si intenteu fer un fetch('fitxer.txt') estant sota aquest protocol, el navegador llançarà un error de seguretat immediat. El navegador fa això perquè detecta que l'HTML està intentant "saltar" directament al teu disc dur per llegir un fitxer local sense passar per un servidor web. El Sandbox bloca per complet el protocol file:// dins de les comandes de xarxa com fetch per evitar que un HTML maliciós descarregat pugui xafardejar i extreure informació privada de les teves carpetes locals.
## Seguretat en l'Entorn Client: SOP i CORS

Quan les aplicacions web necessiten demanar dades a un servidor en segon pla, entren en joc les regles de seguretat més estrictes del navegador.

### La Política del Mateix Origen (Same-Origin Policy - SOP)

És la política de seguretat base del navegador. Estableix que un script d'una pàgina web **només pot demanar dades a un altre recurs si ambdós tenen exactament el mateix origen**.
Què és un "origen"? El navegador calcula l'origen mirant tres elements de la URL:

* **Protocol** (ex: `https`)
* **Domini / Host** (ex: `spotify.com`)
* **Port** (ex: `:443` per defecte a HTTPS, o `:3000` en desenvolupament)

Si un sol d'aquests tres elements canvia, el navegador ho considera un **origen diferent** i, per defecte, bloca la resposta de la petició per protegir l'usuari (evita que una web externa robi dades de la teva sessió oberta en un altra).

### El mecanisme CORS (Cross-Origin Resource Sharing)

Què passa si realment *necessitem* demanar dades entre dos orígens diferents? Això és exactament el que viureu en el projecte de classe: el vostre Frontend (React) s'executarà a `http://localhost:5173` (port 5173) i el vostre Backend (API) s'executarà a `http://localhost:3000` (port 3000). Com que els ports són diferents, **el navegador activarà la SOP i blocarà les peticions de React cap a l'API llançant un error de CORS.**

Per solucionar-ho, **la configuració s'ha de fer sempre al Servidor (Backend)**. El servidor ha d'enviar en les seves respostes unes capçaleres HTTP especials d'autorització. La més important és:
`Access-Control-Allow-Origin: http://localhost:5173`

Quan el navegador de l'usuari veu aquesta capçalera provinent de l'API, entén que el servidor dóna permís explícit al frontend de React per llegir les dades (com les cançons) i llavors, i només llavors, permet que la comunicació tingui èxit i la web funcioni.

[^1]: En el model client/servidor, el navegador necessita demanar coses a la xarxa. JavaScript utilitza una funció nativa anomenada fetch() (que en anglès significa "anar a buscar").
	
	Quan executem fetch('https://api.spotify.com/songs'), el navegador obre un canal de comunicació per xarxa, viatja fins al servidor destí, demana el recurs i ens el retorna.
