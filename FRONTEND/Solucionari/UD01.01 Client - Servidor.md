## Exploració de les capacitats del client mitjançant l'objecte `navigator`

### Resultats de l'execució de comandes a la consola:

En executar les línies de codi a la consola de les DevTools, s'obtenen uns resultats similars a aquests (poden variar segons el dispositiu de l'alumne):

* **`navigator.userAgent`:** `"Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/120.0.0.0 Safari/537.36"`
* **`navigator.language`:** `"ca"` (o `"es-ES"`, `"en-US"` depenent de la configuració del sistema).
* **`navigator.onLine`:** `true` (si l'ordinador està connectat a Internet).

### Resposta a la pregunta sobre el `userAgent`:

El **`userAgent`** és una cadena de text que el navegador envia al servidor a cada petició HTTP per identificar-se. Conté informació detallada sobre el nom del navegador, la seva versió, el motor de renderitzat i el sistema operatiu de l'usuari.

**Utilitat en el desenvolupament web:**

* **Adaptabilitat i Compatibilitat:** Permet al codi de client prendre decisions tècniques o aplicar canvis d'estils si detecta que l'usuari fa servir un navegador antic o un sistema operatiu mòbil específic (iOS/Android).

* **Analítica:** Serveix per recopilar estadístiques sobre quin tipus de públic visita la web i optimitzar l'aplicació per als navegadors més utilitzats pels nostres clients.
## Demostració de les limitacions del Sandbox (Seguretat de fitxers local)

### Error detectat a la consola:

Quan s'obre el fitxer `index.html` mitjançant el protocol local (`file:///`), l'execució del `fetch('dades_secretes.txt')` falla immediatament. A la consola es mostra el següent error en vermell:

> `TypeError: Failed to fetch`
> o també:
> `Access to fetch at 'file:///C:/ruta/dades_secretes.txt' from origin 'null' has been blocked by CORS policy: Cross origin requests are only supported for protocol schemes: http, data, chrome, chrome-extension, chrome-untrusted, https.`

## Exercici de reflexió (Per què el navegador actua així?):

- Per què es bloqueja? El navegador bloqueja l'acció perquè, en estar obrint l'HTML des de l'escriptori (file://), la comanda fetch('dades_secretes.txt') intenta fer servir el motor de xarxa del navegador per accedir directament a l'emmagatzematge local de l'ordinador sense cap intermediari. El Sandbox ho té prohibit per complet.

- Diferència de protocols: Quan fem servir http:// o https://, JavaScript no entra mai al nostre disc dur; simplement fa una petició a través d'Internet a un servidor extern, i és aquest servidor qui decideix de manera segura quins fitxers ens envia i quins no. En canvi, amb file://, el codi intenta saltar-se el servidor i llegir la nostra màquina directament.

- Impacte en la privacitat: Si el Sandbox permetés fer fetch sota el protocol file://, la nostra privacitat estaria totalment compromesa. Qualsevol pàgina web maliciosa que descarreguéssim o obríssim en local podria incloure un codi ocult que executés peticions automatitzades a carpetes privades de l'ordinador (com ara les nostres fotos, documents o fitxers de configuració del sistema operatiu) per extreure'n la informació i enviar-la a un servidor de tercers sense que ens n'adonéssim.
## Provocació simulada d'un error de CORS

### Error llançat pel navegador:

En intentar fer un `fetch('https://www.apple.com')` des de la consola d'un domini completament diferent (com Google o Viquipèdia), el motor de JavaScript denega la resposta i llança el següent error de seguretat:

> `Access to fetch at 'https://www.apple.com/' from origin 'https://www.google.com' has been blocked by CORS policy: No 'Access-Control-Allow-Origin' header is present on the requested resource. If an opaque response serves your needs, set the request's mode to 'no-cors' to fetch the resource with CORS disabled.`

### Diagnòstic del problema i solució de cara al projecte Spotify:

* **Per què passa?** L'error es produeix per la **Política del Mateix Origen (SOP)**. El navegador detecta que el codi (l'origen de la petició) s'està executant a `https://www.google.com`, però està intentant demanar un recurs que pertany a `https://www.apple.com`. Com que els dominis són diferents i Apple no té configurada una autorització explícita per a Google, el navegador talla la comunicació per protegir l'usuari.
* **Com afectarà el nostre projecte?** Quan creem el frontend en **React** (que s'executarà normalment a `http://localhost:5173`) i vulguem demanar la llista de cançons al nostre **Backend/API** (que estarà corrent a `http://localhost:3000`), el navegador ho blocarà amb exactament aquest mateix error perquè els ports són diferents (es consideren orígens diferents).
* **Solució que cal aplicar al Backend:** Per solucionar-ho, no hem de modificar res a React. Hem d'anar al codi del nostre **servidor (API)** i configurar el mecanisme **CORS**. Si estem fent servir Node.js amb Express, haurem d'instal·lar i activar el middleware de CORS, configurant la capçalera de resposta HTTP:
```javascript
// Al backend de Node.js / Express:
res.setHeader('Access-Control-Allow-Origin', 'http://localhost:5173');
```
D'aquesta manera, el servidor li dirà formalment al navegador: *"Sí, dono permís per compartir les meves dades amb l'aplicació React que s'executa a localhost:5173"*, i el navegador permetrà que la interfície de Spotify funcioni correctament.
