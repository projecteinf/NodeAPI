
Sí, es pot millorar força. La idea és bona, però ara mateix el tutorial barreja “aprendre API des de zero” amb “construir API professional” massa ràpid. Això pot funcionar per alumnat avançat, però per estudiants que reprodueixin el projecte pas a pas hi ha punts que els faran encallar.

## Diagnòstic crític

El plantejament general és encertat: començar amb Express, passar a TypeScript, separar capes, afegir base de dades, validació, autenticació i seguretat. Però hi ha alguns problemes importants.

### 1. El salt de dificultat és massa brusc

Passes de rutes en memòria a Prisma, PostgreSQL, JWT, Zod, middleware, arquitectura MVC i testing en molt poc espai. Per a un tutorial docent, això és massa dens.

Millor alternativa: dividir-ho en **mòduls independents**, cadascun amb un objectiu observable:

1. API mínima.
2. API amb dades en memòria.
3. CRUD complet sense base de dades.
4. Persistència amb base de dades.
5. Validació.
6. Errors professionals.
7. Autenticació.
8. Relacions entre recursos.
9. Testing i documentació.
10. Docker / desplegament.

Això permet que cada etapa tingui una pràctica reproduïble i avaluable.

### 2. PostgreSQL és bona opció, però cal decidir-la

Dius MySQL/MariaDB o PostgreSQL, però el tutorial ja està escrit per PostgreSQL. Amb Prisma podries adaptar-ho, però per estudiants és millor **no deixar aquesta decisió oberta**.

Jo triaria **PostgreSQL** per defecte. Prisma té una guia oficial directa per configurar un projecte TypeScript amb PostgreSQL i generar Prisma Client, cosa que encaixa bé amb el teu objectiu docent. ([Prisma][1])

Millor frase:

> En aquest tutorial farem servir PostgreSQL. MySQL/MariaDB també serien possibles amb Prisma, però per evitar dispersió treballarem amb una sola base de dades.

### 3. Node.js v18 ja no és la millor recomanació per a un projecte nou

El teu text diu Node.js v18 o superior. Avui, 13 de maig de 2026, és millor recomanar **Node.js 24 LTS** per a projectes nous. La pàgina oficial de versions mostra Node.js 24 com a LTS, mentre que Node.js 26 és “Current” i no entrarà en LTS fins a l’octubre de 2026. ([Node.js][2])

### 4. Hi ha un error real en la validació amb Zod

Aquest fragment és problemàtic:

```typescript
export const createTrackSchema = z.object({
  name: z.string().min(1).max(100),
  duration: z.number().int().positive(),
  artistId: z.number().int().positive()
});
```

I després fas:

```typescript
await schema.parseAsync({
  body: req.body,
  query: req.query,
  params: req.params
});
```

Això no valida correctament perquè l’esquema espera `name`, `duration` i `artistId` al nivell superior, però tu li passes un objecte amb `body`, `query` i `params`.

Hauria de ser així:

```typescript
import { z } from 'zod';

export const createTrackSchema = z.object({
  body: z.object({
    name: z.string().min(1).max(100),
    duration: z.number().int().positive(),
    artistId: z.number().int().positive()
  })
});

export const trackIdSchema = z.object({
  params: z.object({
    id: z.string().regex(/^\d+$/).transform(Number)
  })
});
```

Això és important perquè Zod és una bona elecció precisament perquè combina validació en temps d’execució amb inferència de tipus TypeScript. ([Zod][3])

### 5. L’autenticació està incompleta

Dius que cal afegir `password` al model `User`, però el model inicial no el té. Això farà que l’alumnat copiï el codi i falli.

El model hauria de néixer ja pensant en autenticació, o bé has de crear una etapa clara de migració.

Millor model inicial si ja saps que hi haurà login:

```prisma
model User {
  id           Int        @id @default(autoincrement())
  email        String     @unique
  name         String
  passwordHash String
  playlists    Playlist[]
  createdAt    DateTime   @default(now())
  updatedAt    DateTime   @updatedAt
}
```

I el servei no hauria de retornar mai l’usuari amb la contrasenya:

```typescript
const { passwordHash, ...safeUser } = user;
return { user: safeUser, token };
```

També eliminaria això:

```typescript
const JWT_SECRET = process.env.JWT_SECRET || 'supersecret';
```

En un tutorial professional és millor fallar explícitament si falta el secret:

```typescript
if (!process.env.JWT_SECRET) {
  throw new Error('JWT_SECRET is required');
}
```

### 6. Express 5 canvia la manera recomanable de tractar errors async

El tutorial fa servir molts `try/catch` dins dels controladors. Això és acceptable amb Express 4, però Express 5 ja gestiona promeses rebutjades en handlers async i les passa al middleware d’errors. La mateixa documentació d’Express 5 destaca aquesta millora. ([expressjs.com][4])

Per a un tutorial modern, jo faria una de dues coses:

Opció docent simple:

```typescript
export const getAllTracks = async (req: Request, res: Response) => {
  const tracks = await trackService.getAllTracks();
  res.json(tracks);
};
```

I deixaria que el middleware global gestioni errors.

Opció més professional:

Crear errors propis:

```typescript
export class AppError extends Error {
  constructor(
    public statusCode: number,
    message: string
  ) {
    super(message);
  }
}
```

I després:

```typescript
if (!track) {
  throw new AppError(404, 'Track not found');
}
```

### 7. Falta una etapa molt important: contracte de l’API

Per a un tutorial professional, jo afegiria abans del codi una taula d’endpoints. Això ajuda molt l’alumnat.

Exemple:

| Method | Endpoint                                 | Description                | Auth      |
| ------ | ---------------------------------------- | -------------------------- | --------- |
| GET    | `/tracks`                                | List tracks                | No        |
| GET    | `/tracks/:id`                            | Get one track              | No        |
| POST   | `/tracks`                                | Create track               | Yes/Admin |
| GET    | `/artists`                               | List artists               | No        |
| POST   | `/playlists`                             | Create playlist            | Yes       |
| POST   | `/playlists/:playlistId/tracks/:trackId` | Add track to playlist      | Yes       |
| DELETE | `/playlists/:playlistId/tracks/:trackId` | Remove track from playlist | Yes       |

Això converteix el tutorial en un projecte reproduïble, no només en una seqüència de fragments.

## Canvi important: afegiria Docker abans que Prisma

Ara dius “instal·la PostgreSQL”. Això és un problema en una aula: cada alumne tindrà un sistema diferent, errors de ports, permisos, usuaris, contrasenyes, serveis que no arrenquen, etc.

Millor:

```yaml
services:
  db:
    image: postgres:16
    container_name: spotify_api_db
    restart: always
    environment:
      POSTGRES_USER: spotify
      POSTGRES_PASSWORD: spotify
      POSTGRES_DB: spotify_db
    ports:
      - "5432:5432"
    volumes:
      - postgres_data:/var/lib/postgresql/data

volumes:
  postgres_data:
```

I al `.env`:

```env
DATABASE_URL="postgresql://spotify:spotify@localhost:5432/spotify_db"
```

Això redueix moltíssim fricció.

## Millora del model de dades

El model actual és massa senzill per a una API estil Spotify. Jo afegiria `Album`, encara que sigui més endavant.

Versió millorada:

```prisma
model User {
  id           Int        @id @default(autoincrement())
  email        String     @unique
  name         String
  passwordHash String
  playlists    Playlist[]
  createdAt    DateTime   @default(now())
  updatedAt    DateTime   @updatedAt
}

model Artist {
  id        Int      @id @default(autoincrement())
  name      String
  albums    Album[]
  tracks    Track[]
  createdAt DateTime @default(now())
}

model Album {
  id        Int      @id @default(autoincrement())
  title     String
  artistId  Int
  artist    Artist   @relation(fields: [artistId], references: [id])
  tracks    Track[]
  createdAt DateTime @default(now())
}

model Track {
  id        Int        @id @default(autoincrement())
  title     String
  duration  Int
  artistId  Int
  albumId   Int?
  artist    Artist     @relation(fields: [artistId], references: [id])
  album     Album?     @relation(fields: [albumId], references: [id])
  playlists Playlist[] @relation("PlaylistTracks")
  createdAt DateTime   @default(now())
}

model Playlist {
  id        Int      @id @default(autoincrement())
  name      String
  userId    Int
  user      User     @relation(fields: [userId], references: [id])
  tracks    Track[]  @relation("PlaylistTracks")
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt
}
```

Això dona més joc didàctic: relacions 1:N, N:M, camps opcionals, timestamps i consultes amb `include`.

1. massa salts de dificultat;
