# 3. Avantatges

## Separació entre client i servidor

El client no necessita saber com està feta la base de dades ni com funciona internament el servidor. Només necessita conèixer les rutes de l'API.

Per exemple:

```http
GET /artists/3
```

El client no sap si les dades venen de PostgreSQL, MySQL, MongoDB o un altre servei. I això és bo.

---

## Ús d'estàndards coneguts

REST utilitza HTTP, que és un protocol molt conegut. Això fa que sigui fàcil d'entendre i de provar.

Els mètodes més habituals són:

| Mètode   | Ús habitual                 |
| -------- | --------------------------- |
| `GET`    | Consultar dades             |
| `POST`   | Crear dades                 |
| `PUT`    | Substituir dades            |
| `PATCH`  | Modificar parcialment dades |
| `DELETE` | Eliminar dades              |

Exemple:

```http
DELETE /playlists/4
```

Aquesta petició indicaria que volem eliminar la playlist amb identificador `4`.

---

## Facilita la reutilització

Una mateixa API pot ser utilitzada per diferents clients:

```text
Aplicació web
Aplicació Android
Aplicació iOS
Aplicació d'administració
Altres serveis externs
```

Tots poden comunicar-se amb el mateix servidor utilitzant les mateixes rutes.

---

## Escalabilitat

REST acostuma a funcionar bé en sistemes grans perquè cada petició HTTP conté la informació necessària per ser processada.

Per exemple, si el servidor rep:

```http
GET /me/playlists
Authorization: Bearer eyJ...
```

La petició ja inclou el token necessari per identificar l'usuari. Això facilita repartir peticions entre diversos servidors.

---

## Facilitat de prova

Una API REST es pot provar fàcilment amb eines com:

```text
curl
Postman
Insomnia
Thunder Client
Navegador web, en alguns casos
```

Per exemple:

```bash
curl http://localhost:3000/tracks
```

Això és molt útil en entorns educatius i professionals.

---

# 4. Inconvenients

## Pot estar mal dissenyada fàcilment

REST sembla senzill, però moltes APIs acaben sent inconsistents.

Un mal exemple seria:

```http
GET /getAllSongs
POST /deleteSong
GET /createUser
```

Això barreja accions amb recursos i fa un mal ús dels mètodes HTTP.

Una versió millor seria:

```http
GET /tracks
DELETE /tracks/15
POST /users
```

La diferència és important: en REST, les rutes representen **recursos**, i els mètodes HTTP representen **accions**.

---

## Pot retornar massa dades o massa poques

Una API REST mal pensada pot obligar el client a fer moltes peticions.

Per exemple, per mostrar una playlist amb les seves cançons i artistes, potser caldria fer:

```http
GET /playlists/1
GET /playlists/1/tracks
GET /tracks/7/artist
GET /tracks/8/artist
GET /tracks/9/artist
```

Això pot ser ineficient. Aquest problema és un dels motius pels quals existeixen alternatives com **GraphQL**.

---

## Versionar l'API pot ser complicat

Quan una API ja està en ús, canviar-la pot trencar aplicacions existents.

Per això sovint es versionen les APIs:

```http
/api/v1/tracks
/api/v2/tracks
```

El problema és que mantenir diverses versions alhora pot augmentar la complexitat del projecte.

---

## REST no defineix tots els detalls

REST dona principis generals, però no imposa una única manera de fer les coses. Això dona flexibilitat, però també pot generar inconsistències.

Per exemple, diferents equips poden tenir criteris diferents sobre:

```text
noms de rutes
format dels errors
paginació
filtres
autenticació
estructura de les respostes
```

Per això en projectes professionals és important definir una guia d'estil de l'API.

---

# 5. Conceptes clau d'una API REST

## Recursos

Un recurs és una entitat del sistema.

En una API musical podríem tenir:

```text
/users
/tracks
/artists
/albums
/playlists
```

Cada recurs sol tenir un identificador:

```http
GET /tracks/25
```

Això demana la cançó amb identificador `25`.

---

## Mètodes HTTP

Els mètodes indiquen què volem fer amb el recurs.

```http
GET /tracks
```

Consultar cançons.

```http
POST /tracks
```

Crear una cançó.

```http
PATCH /tracks/25
```

Modificar parcialment la cançó `25`.

```http
DELETE /tracks/25
```

Eliminar la cançó `25`.

---

## Codis d'estat HTTP

Una API REST hauria d'utilitzar correctament els codis d'estat HTTP.

Alguns dels més habituals són:

| Codi                        | Significat                             |
| --------------------------- | -------------------------------------- |
| `200 OK`                    | La petició s'ha completat correctament |
| `201 Created`               | S'ha creat un nou recurs               |
| `204 No Content`            | Operació correcta sense resposta       |
| `400 Bad Request`           | La petició és incorrecta               |
| `401 Unauthorized`          | Falta autenticació                     |
| `403 Forbidden`             | No tens permisos                       |
| `404 Not Found`             | El recurs no existeix                  |
| `500 Internal Server Error` | Error intern del servidor              |

Exemple:

```http
GET /tracks/999
```

Si la cançó no existeix, la resposta correcta seria:

```http
404 Not Found
```

No seria correcte retornar `200 OK` amb un missatge dient “no trobat”. Això confon el client.

---

## Format de dades

Actualment, el format més habitual és **JSON**.

Exemple de resposta:

```json
{
  "id": 12,
  "title": "One More Time",
  "duration": 320,
  "artist": {
    "id": 4,
    "name": "Daft Punk"
  }
}
```

JSON és llegible, fàcil de generar i compatible amb pràcticament qualsevol llenguatge de programació.

---

# 6. Exemple complet senzill

Suposem una API de playlists.

## Consultar totes les playlists

```http
GET /playlists
```

Resposta:

```json
[
  {
    "id": 1,
    "name": "Rock clàssic"
  },
  {
    "id": 2,
    "name": "Música per estudiar"
  }
]
```

## Consultar una playlist concreta

```http
GET /playlists/1
```

Resposta:

```json
{
  "id": 1,
  "name": "Rock clàssic",
  "tracks": [
    {
      "id": 10,
      "title": "Bohemian Rhapsody"
    },
    {
      "id": 11,
      "title": "Hotel California"
    }
  ]
}
```

## Crear una playlist

```http
POST /playlists
Content-Type: application/json
```

Cos de la petició:

```json
{
  "name": "Cançons per entrenar"
}
```

Resposta:

```http
201 Created
```

```json
{
  "id": 3,
  "name": "Cançons per entrenar"
}
```

## Eliminar una playlist

```http
DELETE /playlists/3
```

Resposta:

```http
204 No Content
```

# 7. Bones pràctiques bàsiques

Una API REST ben dissenyada hauria de seguir alguns criteris:

```text
Utilitzar noms de recursos en plural
Fer servir correctament els mètodes HTTP
Retornar codis d'estat adequats
Validar les dades d'entrada
Tenir una estructura d'errors coherent
Afegir autenticació quan sigui necessari
Documentar les rutes
No exposar detalls interns de la base de dades
```

Per exemple, millor això:

```http
GET /tracks
POST /tracks
GET /tracks/15
PATCH /tracks/15
DELETE /tracks/15
```

Que no pas això:

```http
GET /getTracks
POST /newTrack
POST /updateTrack
GET /deleteTrack
```

La segona opció és més pobra perquè ignora la semàntica d'HTTP.

---

# 8. Resum final

Una **API REST** és una forma estàndard i molt utilitzada perquè aplicacions diferents es comuniquin a través d'HTTP.

Serveix per separar el client del servidor, protegir l'accés a les dades i oferir una interfície clara per consultar, crear, modificar o eliminar recursos.

La idea principal és:

```text
Recursos + mètodes HTTP + codis d'estat + JSON
```

Exemple:

```http
GET /tracks
POST /playlists
PATCH /users/4
DELETE /albums/9
```

REST és una opció molt adequada per aprendre a construir APIs perquè és simple, popular i propera al funcionament real del web. Però no és màgia: una API REST mal dissenyada pot ser confusa, ineficient i difícil de mantenir. El valor real no és només “fer endpoints”, sinó dissenyar una interfície coherent, segura i fàcil d'utilitzar.
